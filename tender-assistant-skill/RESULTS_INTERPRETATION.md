# Как интерпретировать результаты анализа тендера

После запуска skill-а для каждого тендера создаётся отдельная папка результата:

```text
D:\skill_zip_test\out\<Название тендера>\
```

Внутри обычно находятся три файла:

```text
tender_summary.md
questions_for_customer.md
tender_score.json
```

Основной файл для чтения человеком — `tender_summary.md`.  
Файл `questions_for_customer.md` нужен для уточняющих вопросов заказчику.  
Файл `tender_score.json` нужен для детальной проверки и диагностики.

---

## 1. `tender_summary.md`

Это краткая сводка по тендеру.

Открывать в первую очередь:

```powershell
Get-Content "D:\skill_zip_test\out\Тендер 1\tender_summary.md" -Encoding UTF8
```

Или открыть в VS Code / Markdown viewer.

### 1.1. Общий вывод

В начале файла находится общий вывод.

Пример:

```text
По текущим критериям критичные риски не выявлены.
```

Это означает, что по настроенным критериям тендер выглядит подходящим.

Другой пример:

```text
Требуется ручная проверка перед принятием решения по тендеру.
```

Это означает, что автоматическая проверка нашла проблему, риск, противоречие или недостаток данных.

---

## 2. Итоговый сценарий

В разделе `Итоговый сценарий` указывается итоговая классификация тендера.

Пример:

```text
Сценарий: relevant_dealer
Рекомендация: Передать тендер дилеру/партнёру
Требуется ручная проверка: нет
```

Интерпретация:

```text
Тендер релевантен.
Критичных проблем не найдено.
Можно передавать дилеру или партнёру.
```

Пример:

```text
Сценарий: need_human_review
Требуется ручная проверка: да
```

Интерпретация:

```text
Тендер нельзя принимать автоматически.
Нужно вручную проверить проблемные критерии.
```

---

## 3. Основные сценарии

### `relevant_dealer`

Тендер подходит по текущим критериям, но может быть удобнее передать его дилеру или партнёру.

Обычно это нормальный положительный сценарий.

Что делать:

```text
Передать тендер дилеру/партнёру или продолжить внутреннюю проверку по бизнес-процессу.
```

### `need_human_review`

Требуется ручная проверка.

Причины могут быть разные:

```text
- не найдено подтверждение важного критерия;
- найден другой способ закупки;
- найден средний или высокий риск;
- есть противоречие;
- не хватает данных в документах.
```

Что делать:

```text
Открыть раздел "Критерии, требующие внимания" и проверить указанные пункты вручную.
```

---

## 4. Поле `Требуется ручная проверка`

```text
Требуется ручная проверка: нет
```

Означает:

```text
Автоматическая проверка не нашла критериев, требующих обязательной ручной проверки.
```

```text
Требуется ручная проверка: да
```

Означает:

```text
Есть хотя бы один критерий, который нельзя принять автоматически.
Нужно посмотреть проблемные критерии и evidence.
```

Важно: `ручная проверка: да` не всегда означает, что тендер плохой.  
Иногда это значит, что в документах просто не найдено явное подтверждение.

---

## 5. Блокирующие критерии

В summary может быть блок:

```text
Блокирующие критерии:
- procurement_method — status: fail, risk: high
```

Это значит, что найден критерий, который влияет на итоговое решение.

Основные значения `status`:

```text
pass      — критерий подтверждён
fail      — критерий не пройден
unknown   — данных недостаточно
conflict  — найдены противоречивые данные
```

Основные значения `risk`:

```text
low       — низкий риск
medium    — средний риск
high      — высокий риск
```

---

## 6. Как читать статусы критериев

### `pass`

Критерий подтверждён.

Пример:

```text
purchase_type_goods — Закупка является поставкой товара
```

Интерпретация:

```text
В документах найдено подтверждение, что закупка относится к поставке товара.
```

### `fail`

Критерий не пройден.

Пример:

```text
procurement_method — status: fail
```

Если в evidence указано:

```text
запрос предложений в электронной форме
```

а критерий ожидает:

```text
электронный аукцион
```

интерпретация такая:

```text
Способ закупки не соответствует критерию.
Нужно вручную проверить допустимость участия.
```

### `unknown`

Критерий не подтверждён.

Пример:

```text
procurement_method — status: unknown
```

Интерпретация:

```text
В документах не найдено явное подтверждение способа закупки.
Это не автоматический отказ, но требуется ручная проверка.
```

### `conflict`

Найдены противоречивые признаки.

Интерпретация:

```text
В разных местах документов найдены признаки, которые противоречат друг другу.
Нужно вручную сверить документы.
```

---

## 7. Раздел `Критерии, требующие внимания`

Этот раздел важнее общей статистики.

В нём перечислены критерии, которые требуют проверки человеком.

Пример:

```text
ID критерия: procurement_method
Критерий: Способ закупки — электронный аукцион
Статус: fail
Риск: high
Комментарий: Найден другой способ закупки, а не электронный аукцион.
```

Интерпретация:

```text
Автоматическая проверка нашла серьёзную проблему.
Нужно проверить способ закупки в извещении и документации.
```

Пример:

```text
Статус: unknown
Комментарий: Подтверждение способа закупки как электронный аукцион не найдено.
```

Интерпретация:

```text
Skill не нашёл явного подтверждения.
Нужно проверить документы вручную.
```

---

## 8. Evidence / подтверждающие фрагменты

`Evidence` — это фрагменты текста из документов, на основании которых skill сделал вывод.

Пример:

```text
способ осуществления закупки | запрос предложений в электронной форме
```

Интерпретация:

```text
В документах явно найден способ закупки "запрос предложений".
Если критерий ожидает "электронный аукцион", это проблема.
```

Evidence нужно использовать так:

```text
1. Проверить, что фрагмент действительно относится к нужному критерию.
2. Найти этот фрагмент в исходном документе.
3. Подтвердить или опровергнуть вывод skill-а вручную.
```

---

## 9. Подтверждённые критерии

Раздел показывает критерии, которые skill считает пройденными.

Пример:

```text
subject_title_oil — Название закупки содержит товарную сущность «масло»
```

Интерпретация:

```text
Предмет закупки действительно связан с маслами или смазочными материалами.
```

Важно: этот раздел не всегда нужно читать полностью.  
Для принятия решения важнее:

```text
- общий вывод;
- итоговый сценарий;
- блокирующие критерии;
- критерии, требующие внимания.
```

## 10. `questions_for_customer.md`

Этот файл содержит вопросы, которые стоит задать заказчику или ответственному сотруднику.

Открыть:

```powershell
Get-Content "D:\skill_zip_test\out\Тендер 1\questions_for_customer.md" -Encoding UTF8
```

Как использовать:

```text
1. Если файл пустой или содержит сообщение, что вопросов нет — явных уточнений не требуется.
2. Если есть вопросы — их нужно проверить перед принятием решения.
3. Вопросы обычно связаны с неподтверждёнными, спорными или рискованными критериями.
```

---

## 11. `tender_score.json`

Это технический файл с полной структурой результата.

Открывать его нужно, если:

```text
- summary недостаточно;
- нужно проверить все критерии;
- нужно увидеть полный evidence;
- нужно проверить работу LLM;
- нужно разобраться, почему тендер попал в need_human_review.
```

Основные поля:

```text
rules
scenario_result
stats
document_count
criteria_count
input_path
```

---

## 12. Как читать `scenario_result` в JSON

Пример:

```json
"scenario_result": {
  "scenario": "need_human_review",
  "human_review_required": true,
  "confidence": "medium"
}
```

Интерпретация:

```text
Итоговый сценарий — ручная проверка.
Уверенность средняя.
Автоматически принимать решение нельзя.
```

---

## 13. Как читать `rules[]` в JSON

Каждый элемент `rules[]` — это результат проверки одного критерия.

Ключевые поля:

```text
id                       — внутренний ID критерия
block                    — блок критерия
criterion                — текст критерия
status                   — результат проверки
risk                     — уровень риска
priority                 — приоритет критерия
human_review_required    — требуется ли ручная проверка
comment                  — комментарий
evidence                 — подтверждающие фрагменты
llm_verdict              — результат LLM-проверки, если она применялась
```

---

## 14. Как читать `llm_verdict`

LLM-проверка используется не всегда. В selective-режиме она вызывается только там, где нужна дополнительная проверка.

Главное поле:

```text
rules[].llm_verdict.invocation_status
```

Возможные значения:

```text
ok            — LLM была вызвана и вернула результат
skipped       — LLM не вызывалась, потому что rule-based результата достаточно
unavailable   — LLM недоступна или timeout
error         — ошибка при вызове LLM
invalid_json  — LLM вернула некорректный JSON
```

---

## 15. Как интерпретировать `skipped`

Пример:

```json
"invocation_status": "skipped",
"verdict": "pass",
"reason": "Deterministic rule is pass/low/no-review; LLM shadow skipped by selective mode."
```

Интерпретация:

```text
LLM не вызывалась.
Обычная rule-based проверка уже дала уверенный результат.
Это нормальное поведение.
```

---

## 16. Как интерпретировать `ok`

Пример:

```json
"invocation_status": "ok",
"verdict": "fail",
"reason": "В evidence указан способ закупки 'запрос предложений', что не соответствует критерию 'электронный аукцион'."
```

Интерпретация:

```text
LLM была вызвана и подтвердила проблему.
В этом случае нужно смотреть verdict, reason и supporting_evidence_ids.
```

---

## 17. Как интерпретировать `unavailable`

```text
unavailable > 0
```

Означает:

```text
LLM не смогла отработать.
Возможные причины: Ollama не запущена, модель не загружена, timeout слишком маленький.
```

Что проверить:

```powershell
Invoke-RestMethod http://localhost:11434/api/tags
ollama list
```

Также проверить `.env`:

```env
TENDER_LLM_TIMEOUT_SECONDS=300
```

## 18. Быстрый контрольный скрипт

Для проверки всех `tender_score.json`:

```powershell
@'
from pathlib import Path
import json
import sys

sys.stdout.reconfigure(encoding="utf-8", errors="replace")

root = Path(r"D:\skill_zip_test\out")
score_files = sorted(root.rglob("tender_score.json"))

print("score files:", len(score_files))

for score_path in score_files:
    data = json.loads(score_path.read_text(encoding="utf-8"))

    counts = {
        "ok": 0,
        "skipped": 0,
        "unavailable": 0,
        "error": 0,
        "invalid_json": 0,
    }

    procurement_method_verdict = None

    for rule in data.get("rules", []):
        verdict = rule.get("llm_verdict")
        if not verdict:
            continue

        status = verdict.get("invocation_status")
        counts[status] = counts.get(status, 0) + 1

        if rule.get("id") == "procurement_method":
            procurement_method_verdict = verdict

    scenario = data.get("scenario_result", {})

    print()
    print("folder:", score_path.parent)
    print("scenario:", scenario.get("scenario"))
    print("human_review_required:", scenario.get("human_review_required"))
    print("llm_counts:", counts)
    print("procurement_method:", procurement_method_verdict)
'@ | python
```

## 19. Как принять итоговое решение

### Можно передавать дальше

Если:

```text
scenario: relevant_dealer
human_review_required: False
fail: 0
conflict: 0
high risk: 0
```

Решение:

```text
Тендер можно передавать дилеру/партнёру или продолжать обработку.
```

### Нужна ручная проверка

Если:

```text
scenario: need_human_review
human_review_required: True
```

Решение:

```text
Открыть tender_summary.md.
Проверить блокирующие критерии.
Проверить evidence.
При необходимости открыть исходные документы.
```

### Есть вероятный блокер

Если:

```text
status: fail
risk: high
```

Решение:

```text
Не принимать тендер автоматически.
Проверить критерий вручную.
Если fail подтверждается — не участвовать или передать на отдельное решение.
```

## 20. Пример интерпретации трёх тестовых тендеров

### Тендер 1

```text
scenario: relevant_dealer
human_review_required: False
```

Интерпретация:

```text
Тендер релевантен.
Критичных рисков не найдено.
Ручная проверка по текущим критериям не требуется.
```

### Тендер 2

```text
scenario: need_human_review
procurement_method: unknown
```

Интерпретация:

```text
Нужно вручную проверить способ закупки.
Skill не нашёл явного подтверждения, что это электронный аукцион.
Это не автоматический отказ, а запрос на проверку.
```

### Тендер 3

```text
scenario: need_human_review
procurement_method: fail
```

Интерпретация:

```text
Найден другой способ закупки: запрос предложений.
Критерий "электронный аукцион" не пройден.
Нужна ручная проверка допустимости участия.
```
