# DOCX template mapping

## 1. Цель

Документ фиксирует mapping для будущего DOCX-exporter, который должен формировать `tender_summary.docx` по DOCX-шаблону сводки. Код в рамках этого шага не реализуется.

## 2. Источники анализа

Прочитанные источники:

- DOCX-шаблон: `sources_info/Шаблон сводки по тендеру.docx`;
- `tender-assistant-skill/run.py`;
- `tender-assistant-skill/src/output/summary_writer.py`;
- `tender-assistant-skill/src/output/questions_writer.py`;
- `tender-assistant-skill/src/scoring/scenario_classifier.py`;
- `tender-assistant-skill/src/scoring/rule_engine.py`;
- `tender-assistant-skill/config/criteria.yaml`;
- `tender-assistant-skill/scripts/regression_check.py`.

Для проверки фактической структуры output дополнительно был сгенерирован и прочитан probe для `sources_info/Тендер 1`:

- `outputs/debug/docx_mapping_probe/tender_1/tender_score.json`;
- `outputs/debug/docx_mapping_probe/tender_1/tender_summary.md`;
- `outputs/debug/docx_mapping_probe/tender_1/questions_for_customer.md`.

Probe-файлы являются временными debug-артефактами и не должны коммититься.

## 3. Структура DOCX-шаблона

DOCX-шаблон содержит один основной документ без header/footer parts. Все заголовки в `word/document.xml` выглядят как обычные абзацы без heading-style, поэтому для автоматического заполнения их лучше искать по тексту и/или по порядку таблиц.

Порядок блоков:

1. `ШАБЛОН СВОДКИ ПО ТЕНДЕРУ`
2. `ОБЩАЯ ИНФОРМАЦИЯ`
3. Таблица 1
4. `ЛОТЫ`
5. Таблица 2
6. Текстовое примечание о повторении блока строк для каждого лота
7. `УСЛОВИЯ ПОСТАВКИ`
8. Таблица 3
9. `КОММЕРЧЕСКИЕ УСЛОВИЯ`
10. Таблица 4
11. `ТРЕБОВАНИЯ К УЧАСТНИКУ`
12. Таблица 5
13. `АНАЛИЗ РИСКОВ`
14. Таблица 6
15. `ДОПОЛНИТЕЛЬНАЯ ИНФОРМАЦИЯ`
16. Таблица 7

Явные machine-readable плейсхолдеры не обнаружены.

Также не обнаружены:

- поля вида `{{...}}`;
- квадратные маркеры вида `[...]`;
- подчёркнутые пустые места;
- явные текстовые placeholder tokens.

Таблицы шаблона:

| № | Раздел | Колонки | Строки / поля для заполнения |
|---|---|---|---|
| 1 | `ОБЩАЯ ИНФОРМАЦИЯ` | `Параметр`, `Значение`, `Примечание` | Наименование закупки, заказчик, тип закона, номер закупки / ИКЗ, способ определения поставщика, площадка, сроки, НМЦК, валюта, статус МСП / нацрежим, регион поставки, источник финансирования, срок исполнения контракта |
| 2 | `ЛОТЫ` | `Параметр`, `Значение`, `Примечание` | Наименование позиции, КТРУ / ОКПД2, единица измерения, количество, цена, стоимость, характеристики товара, качество / ГОСТ, срок годности, упаковка, неизменяемые характеристики, страна происхождения / нацрежим |
| 3 | `УСЛОВИЯ ПОСТАВКИ` | `Параметр`, `Значение`, `Примечание` | Адреса поставки, срок поставки, график, логистика / ПРР, ограничения доставки, толеранс объема, подтверждение заявки |

| 5 | `ТРЕБОВАНИЯ К УЧАСТНИКУ` | `Параметр`, `Значение`, `Примечание` | Требования по ст. 31 44-ФЗ, состав заявки, учредительные документы, декларация, национальный режим, подтверждение происхождения |
| 6 | `АНАЛИЗ РИСКОВ` | `Параметр`, `Оценка`, `Комментарий` | Противоречия, договор, формальные требования, логистика, финансы, ценовая конкуренция, сроки, рекомендация к участию |
| 7 | `ДОПОЛНИТЕЛЬНАЯ ИНФОРМАЦИЯ` | `Параметр`, `Значение`, `Примечание` | Ссылка на площадку, контакты заказчика, разъяснения, протокол разногласий, многолотовость, альтернативные предложения, односторонний отказ |

## 4. Доступные данные pipeline

Фактический `tender_score.json` / `score_result` содержит top-level поля:

- `criteria_count`;
- `rules_count`;
- `rules`;
- `stats`;
- `input_path`;
- `document_count`;
- `source_stats`;
- `scenario_result`.

В фактическом output нет top-level поля `evidence`: evidence хранится внутри каждого элемента `rules[]`.

Структура `rules[]` по `rule_engine.py`:

- `id`;
- `block`;
- `criterion`;
- `priority`;
- `evidence`;
- `status`;
- `risk`;
- `human_review_required`;
- `comment`.

Фрагменты `rule.evidence[]` могут использовать поля `snippet`, `text` или `block.text`. В writer-логике evidence уже нормализуется и обрезается: `summary_writer.py` показывает до 2 фрагментов для критериев внимания и 1 фрагмент для подтверждённых критериев, `questions_writer.py` показывает до 3 фрагментов.

Структура `scenario_result`:

- `scenario`;
- `recommendation`;
- `confidence`;
- `reasons`;
- `blocking_criteria`;
- `human_review_required`;
- `stats`.

Возможные сценарии:

- `not_relevant`;
- `relevant_direct`;
- `relevant_dealer`;
- `need_human_review`.

`tender_summary.md` содержит разделы:

- `## 1. Общий вывод`;
- `## 2. Итоговый сценарий`;
- `## 3. Краткая статистика`;
- `## 4. Критерии, требующие внимания`;
- `## 5. Подтверждённые критерии`;
- `## 6. Неподтверждённые низкоприоритетные критерии`.

`questions_for_customer.md` содержит:

- заголовок `# Вопросы заказчику / человеку`;
- `Всего вопросов: N`, если вопросы есть;
- по каждому вопросу: блок, критерий, priority, status, risk, question text, reason/comment, evidence;
- fallback `Вопросы не выявлены.`, если вопросов нет.

Текущий pipeline не выделяет отдельное поле purchase title. | Желательно | `Не извлечено автоматически. См. evidence по subject_title_oil.` |
| Таблица 1: `Заказчик` | `score_result.rules[].evidence` | Текущий deterministic output не содержит отдельного поля customer. Можно искать только вручную по evidence, но автоматическое заполнение без отдельного extractor рискованно. | Опционально | `Не извлечено автоматически.` |
| Таблица 1: `Тип закона` | `score_result.rules[].evidence` | Отдельного поля нет. Заполнять только при явном evidence в будущем extractor. | Опционально | `Нет данных.` |
| Таблица 1: `Номер закупки / ИКЗ` | `score_result.rules[].evidence` | Отдельного поля нет. Не заполнять эвристически в MVP, если нет отдельного extractor. | Опционально | `Не извлечено автоматически.` |
| Таблица 1: `Способ определения поставщика` | `rule id=procurement_method` | Использовать `status`, `comment`, evidence по критерию `procurement_method`. При `pass` можно писать `Электронный аукцион / способ подтверждён evidence`. | Желательно | `Способ закупки не подтверждён.` |

| Таблица 1: `Срок подачи заявок` | `score_result.rules[].evidence` | Отдельного критерия и поля нет. Для MVP оставить пустым или `Нет данных`. | Опционально | `Нет данных.` |
| Таблица 1: `Дата аукциона / рассмотрения заявок` | `score_result.rules[].evidence` | Отдельного поля нет. | Опционально | `Нет данных.` |
| Таблица 1: `НМЦК` | `rule id=price_nmc` | Использовать evidence по `price_nmc`; в `Значение` писать краткий найденный фрагмент или нормализованную цену после будущего extractor. | Желательно | `НМЦК не извлечена.` |
| Таблица 1: `Валюта` | `rule id=price_nmc` | Использовать evidence по `price_nmc`; если явно найдено `руб`/`рубли`, писать `Рубль РФ`. | Желательно | `Нет данных.` |

Если `msp_restriction` is `pass` и scenario `relevant_dealer`, указать признак МСП / работа через дилера. | Обязательно для текущего scoring | `МСП / нацрежим не подтверждены evidence.` |

Заполнять только при явном evidence или оставить как ручные поля. | Опционально | `Нет данных.` |

## 6. Рекомендуемые изменения шаблона, если нужны

Шаблон можно заполнять без изменения файла только по порядку таблиц и названиям строк, но это хрупко: любые ручные изменения таблиц сломают mapping.

Минимальные рекомендуемые изменения шаблона перед реализацией:

- добавить machine-readable placeholders в ячейки `Значение` / `Оценка`, например `{{scenario.recommendation}}`, `{{rules.delivery_period.evidence}}`;
- добавить отдельный раздел `Итоговый сценарий`, потому что сейчас итоговая рекомендация помещается только в строку `Рекомендация к участию` таблицы `АНАЛИЗ РИСКОВ`;
- добавить отдельную таблицу `Критерии внимания` с колонками `ID`, `Блок`, `Критерий`, `Статус`, `Риск`, `Комментарий`, `Evidence`;
- добавить секцию `Вопросы для заказчика`, чтобы не терять содержимое `questions_for_customer.md`;
- добавить секцию `Evidence / основания` или ограниченный appendix, если в DOCX нужно сохранять проверяемость выводов.

## 7. Предлагаемый контракт будущего docx_summary_writer.py

Возможный модуль:

- `tender-assistant-skill/src/output/docx_summary_writer.py`

Возможные функции без реализации:

```python
def render_docx_summary(
    score_result: dict,
    template_path: str | Path,
    output_path: str | Path,
) -> Path:
    ...


def write_docx_summary(
    score_result: dict,
    template_path: str | Path,
    output_path: str | Path,
) -> Path:
    ...
```

Входы:

- `score_result: dict`;
- `template_path: str | Path`;
- `output_path: str | Path`.

Выход:

- `tender_summary.docx`.

Требования к реализации:

- deterministic-only;
- без LLM;
- без изменения исходного DOCX-шаблона;
- каждый автоматически заполненный вывод должен иметь связь с `evidence` или с `scenario_result`;
- если данных нет, использовать явный fallback, а не выдумывать значение;
- длинные evidence-фрагменты обрезать по правилам, совместимым с `summary_writer.py` / `questions_writer.py`.

## 8. Риски и вопросы перед реализацией

- В шаблоне нет плейсхолдеров, поэтому заполнение по порядку таблиц и текстам строк будет хрупким.
- Значительная часть строк шаблона требует структурированных полей, которых текущий `score_result` не содержит: customer, law type, procurement number, platform, submission deadline, contacts, payment terms.
- Если сохранять форматирование DOCX важно, `python-docx` упростит работу с таблицами, но добавлять зависимость можно только отдельным решением. Стандартный `zipfile` + XML возможен, но сложнее для безопасной правки OOXML.
- Нужно заранее определить, какие поля являются обязательными для MVP: минимально `scenario_result`, статистика, criteria/rules risks, вопросы и evidence.
- Нужно ограничивать evidence по длине, иначе таблицы DOCX станут нечитаемыми.
- Нужно определить лимиты строк: сколько критериев внимания, подтверждённых критериев и вопросов включать в основной DOCX.
- Для таблицы `ЛОТЫ` текущий pipeline не строит массив лотов, поэтому MVP может заполнить только один агрегированный блок по найденным evidence.
- Для таблицы `АНАЛИЗ РИСКОВ` часть оценок можно получить из scoring, но ценовая конкуренция и логистическая выполнимость в полном смысле не рассчитываются.

## 9. Итоговая рекомендация

DOCX-exporter можно реализовать без изменения шаблона как MVP: заполнять существующие таблицы по фиксированному порядку и названиям строк, а недоступные поля оставлять с явными fallback-значениями.

Минимально рискованный вариант для MVP: сначала подготовить версию шаблона с machine-readable placeholders и отдельными секциями `Итоговый сценарий`, `Критерии внимания`, `Вопросы для заказчика`, `Evidence / основания`. Это снизит риск поломки mapping при изменении DOCX и позволит сохранить главный принцип проекта: каждый вывод связан с evidence.
