# ADR-0001: LLM runtime для tender assistant

## Status

Accepted

Date: 2026-07-05

## Контекст

Проект tender assistant использует deterministic pipeline для первичной оценки тендеров.

LLM runtime зафиксирован как вспомогательный слой для узких evidence-based задач. На момент принятия ADR runtime ещё не был рабочим слоем обработки, поэтому часть текста ниже сохраняет исторический контекст решения.

LLM-вызов получает:
- один критерий;
- ограниченный список найденных фрагментов `evidence`;
- инструкцию вернуть структурированный JSON.

LLM не должен:
- анализировать весь тендер одним prompt;
- принимать итоговое решение об участии;
- заменять deterministic scoring;
- генерировать выводы без опоры на `evidence`.

Итоговый сценарий тендера выбирают deterministic-компоненты:
- `rule_engine`;
- `scenario_classifier`.

## Решение

Для MVP выбран локальный runtime через Ollama.

Модель должна вызываться через OpenAI-compatible endpoint.

Текущая demo-конфигурация selective mode:

    TENDER_LLM_PROVIDER=ollama
    TENDER_LLM_BASE_URL=http://localhost:11434/v1
    TENDER_LLM_MODEL=qwen2.5:14b
    TENDER_LLM_API_KEY=ollama

## Обоснование

Локальная модель через Ollama выбрана, потому что:
- бесплатна для локального использования;
- не требует платного API;
- подходит для локального MVP;
- поддерживает русскоязычные тендерные документы;
- достаточно компактен для локальных экспериментов;
- может быть заменён позже без изменения архитектуры pipeline.

## Альтернативы

- `qwen3:1.7b` — легче, но потенциально ниже качество.
- `qwen3:8b` — потенциально выше качество, но выше требования к железу.
- Claude / Kimi / OpenAI API — не используются в MVP, потому что проекту нужен бесплатный локальный runtime.
- Полностью rule-based подход — недостаточен для неоднозначной классификации `evidence`.
- Обучение собственной LLM — вне scope MVP.

## Последствия

- Проект не зависит от платного внешнего API.
- Модель можно будет заменить через переменные окружения.
- LLM-слой должен валидировать JSON-ответы.
- Deterministic pipeline остаётся ответственным за итоговые решения.
- LLM-output должен трактоваться как классификация `evidence`, а не как финальная бизнес-логика.

## Текущее состояние реализации

- LLM runtime уже реализован.
- Он работает в advisory/shadow режиме: модель помогает проверить спорные места, но не принимает итоговое решение.
- Selective shadow mode реализован.
- Compact evidence payload реализован: LLM получает короткий набор `evidence`, а не весь тендер.
- Локальный Ollama endpoint поддержан через OpenAI-compatible API.
- В selective mode используется demo-модель `qwen2.5:14b`.
- Deterministic pipeline остаётся источником итогового решения.
