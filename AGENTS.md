# Project instructions

Постоянные правила для AI-агентов, работающих с репозиторием `nadeykin_project`.

## Language

- Отвечать пользователю на русском языке.
- Код, имена файлов, функций, классов, модулей, CLI-команд и технические идентификаторы писать на английском.
- Не переводить имена файлов, путей, функций, классов, модулей и публичных API.

## Project goal

Проект предназначен для анализа тендерных документов, извлечения требований, критериев, рисков и подготовки краткой сводки.

Основная цель проекта — локальный skill/пайплайн тендерного ассистента для первичной скоринговой оценки тендеров.

Ожидаемый пайплайн:

```text
Документы тендера
        ↓
Извлечение текста / таблиц
        ↓
Нормализация в JSON
        ↓
Поиск фактов по критериям
        ↓
Скоринг релевантности / риска
        ↓
Сводка на 1–2 А4
        ↓
Список вопросов человеку / заказчику
```

Ключевой принцип проекта: каждый вывод должен быть связан с найденным `evidence`.

## General principles

- Работай строго в рамках текущей задачи.
- Не расширяй scope самостоятельно.
- Не делай сопутствующие улучшения, если они прямо не запрошены.
- Не исправляй unrelated issues.
- Не меняй архитектуру, публичные API, имена файлов, функций, классов или модулей без явного требования.
- Не гадай по структуре проекта, форматам данных, сигнатурам функций и назначению файлов.
- Перед изменением кода сначала прочитай релевантные файлы.
- Если задача требует анализа принятого файла — сначала сверься с фактическим содержимым файла, а не с предположением из prompt.

## Git restrictions

Запрещено без отдельного явного разрешения:

- создавать git commit;
- выполнять git push;
- создавать pull request;
- менять текущую ветку;
- выполнять rebase;
- выполнять merge;
- выполнять reset;
- выполнять cherry-pick;
- изменять историю Git;
- удалять или переименовывать ветки.

Если задача требует commit, push, PR, merge, rebase, reset или смены ветки — остановись и запроси подтверждение.

## Scope restrictions

- Изменяй только файлы, явно указанные в текущей задаче.
- Если указан один файл — меняй только этот файл.
- Если для решения требуется изменить другой файл — остановись, объясни причину и дождись подтверждения.
- Внутри разрешённого файла меняй только те участки, которые относятся к задаче.
- Не меняй код «по пути».
- Не добавляй новые файлы без прямого указания.
- Не удаляй файлы без прямого указания.
- Не переименовывай файлы без прямого указания.
- Не меняй структуру проекта без прямого требования.

## Minimal diff

- Делай минимально необходимое изменение.
- Не форматируй весь файл.
- Не запускай массовое автоформатирование.
- Не запускай массовые auto-fix команды.
- Не переставляй импорты, если это не требуется для задачи.
- Не меняй стиль кода вне редактируемого участка.
- Не меняй комментарии и документацию, если задача этого не требует.
- Не меняй пробелы, переносы строк и форматирование вне зоны правки.
- Не меняй generated/cache/build артефакты.

## Safety

- Не изменяй `.env`, секреты, токены, ключи и пароли.
- Не выводи секреты, токены, ключи и пароли в ответ.
- Не запускай destructive-команды без явного подтверждения.
- Не удаляй файлы без подтверждения.
- Не выполняй команды, которые могут необратимо изменить данные, без явного подтверждения.
- Не изменяй `.gitignore`, если задача этого явно не требует.
- Не меняй настройки IDE, если задача этого явно не требует.
- Не оставляй временные, debug или backup-файлы.

## Dependencies and environment

- Не добавляй внешние зависимости без явного требования.
- Не меняй `pyproject.toml`, `requirements.txt`, lock-файлы и конфиги окружения без прямого указания.
- Не устанавливай пакеты глобально.
- Не меняй версию Python или настройки виртуального окружения.
- Если зависимость кажется необходимой — остановись и объясни, зачем она нужна.
- Если в окружении отсутствует библиотека, сначала используй существующий fallback проекта, если он предусмотрен.
- Если fallback отсутствует или небезопасен — остановись и объясни проблему.

## Accepted files

Следующие файлы считаются принятыми и не должны меняться без отдельного явного задания:

- `tender-assistant-skill/config/criteria.yaml`
- `tender-assistant-skill/src/normalize/tender_document.py`
- `tender-assistant-skill/src/retrieval/keyword_search.py`
- reader-модули в `tender-assistant-skill/src/ingest/`, если задача не относится к конкретному reader-файлу

Если задача касается одного из этих файлов:

- меняй только указанный файл;
- меняй только описанную часть;
- не делай рефакторинг соседнего кода.

## Before changing code

Перед внесением изменений:

- прочитай файл или файлы, указанные в задаче;
- найди релевантные функции, классы и места использования;
- сверь фактическую структуру кода с описанием в задаче;
- проверь реальные сигнатуры функций;
- проверь реальные форматы входных и выходных данных;
- проверь цепочку вызовов, если задача касается поведения функции;
- убедись, что изменение не ломает downstream-логику внутри разрешённого scope.

Если структура кода отличается от описания задачи:

- остановись;
- кратко опиши расхождение;
- не пытайся угадать решение;
- не расширяй scope без подтверждения.

## Ambiguity handling

Если задача неоднозначна:

- остановись;
- опиши, что именно неоднозначно;
- предложи варианты решения;
- укажи последствия каждого варианта;
- дождись выбора варианта.

Не выбирай самостоятельно вариант, который может изменить архитектуру, API, поведение пайплайна или формат данных.

## Workflow

- Делай небольшие, проверяемые изменения.
- После изменений запускать доступные проверки.
- Если тестов нет, явно указать, что тесты не запускались.
- Не запускать тяжёлые или долгие проверки без необходимости.
- Не запускать весь проект, если задача требует точечной проверки.
- Если проверка не может быть выполнена — явно объяснить почему.
- Если команда завершилась ошибкой — показать релевантный вывод и не скрывать ошибку.

## Verification after changes

После внесения изменений:

- выполни только проверки, указанные в задаче;
- покажи команды, которые запускались;
- покажи релевантный вывод команд;
- покажи diff изменённых файлов;
- покажи статус рабочего дерева.

Минимально в конце работы нужно показать:

```bash
git diff -- <changed-file>
git status --short
```

Если изменено несколько явно разрешённых файлов:

```bash
git diff -- <file-1> <file-2>
git status --short
```

Если проверки не запускались или завершились ошибкой — не утверждай, что задача полностью выполнена.

## Error handling

Если команда завершилась ошибкой:

- не скрывай ошибку;
- покажи релевантный вывод;
- объясни причину, если она понятна;
- не делай дополнительные правки вне scope без подтверждения.

Если ошибка не относится к текущей задаче:

- зафиксируй её отдельно;
- не исправляй её автоматически.

## Tender assistant rules

- `.sign` файлы не анализировать как содержательные документы.
- `sources_info/` считать основным источником реальных тендерных документов.
- `tender-assistant-skill/examples/` считать будущими demo/test fixtures, если задача явно не говорит иное.
- Отсутствие второстепенных данных не считать автоматическим `fail`.
- Противоречия и высокий риск ошибки должны вести к `human_review_required`.
- Модель, если используется, должна работать только на коротких фрагментах `evidence`, а не на всём тендере.
- Каждый вывод должен быть связан с evidence.
- MVP сначала строится детерминированно: readers → normalizer → keyword retrieval → criteria YAML → rule/scoring layer.
- LLM-классификация — отдельный слой и не должна добавляться в модуль, если задача этого явно не требует.
- Итоговые сценарии (`not_relevant`, `relevant_llk`, `relevant_dealer`, `need_human_review`) реализовывать только в модуле, для которого это явно поставлено задачей.

## Final response format

В финальном ответе укажи:

- что было изменено;
- какие файлы изменены;
- какие проверки выполнены;
- результат проверок;
- есть ли незапланированные изменения;
- что показывает `git status --short`;
- был ли создан commit, push или pull request.

Не утверждай, что задача полностью выполнена, если проверки не запускались или завершились ошибкой.

## Hard rule

Если есть конфликт между текущей задачей и этим файлом:

- следуй более строгому ограничению;
- при сомнении остановись и запроси уточнение.
